# v0.1.11

# v0.1.7

## 规章查询面板 UX 大改进

围绕"友好方便、信息清晰"，对规章查询面板做了系统性 UX 重构，解决了反馈最多的几个"用着不顺手"问题。

- **统一文件数字显示**：之前顶部仪表板显示 "1407 已索引"、下方却又有 "本地已索引 2951 篇" 一条独立 badge，两个数字差异让人困惑（实际 2951 是 Tantivy 索引含软删除文档的物理计数，1407 才是数据库逻辑文件数）。现在只保留顶部 `dbSyncStatus.indexed` 作为唯一数字来源，下方改成"● 本地索引就绪"绿点指示器，不再有矛盾数字让用户看花眼。
- **文件夹列表空态引导**：之前 `scanFolders.length === 0` 时整个文件夹列表区直接隐藏，用户加完文件夹前完全不知道"扫描已选的文件夹"按钮要扫哪些。现在空时显示一张引导卡片（📂 + "还没添加扫描文件夹"标题 + 提示文字告诉用户先点上方按钮选择），始终给用户清晰反馈。
- **「添加并扫描」单按钮替代双按钮**：之前界面上有两个语义重叠的按钮 — "添加扫描文件夹"和"扫描已选文件夹"，新手第一次用根本分不清要先点哪个。现在合并成单按钮 `📁 添加并扫描本地文件夹`，选完文件夹自动开始扫描，无需第二次确认；如果之后想批量重扫所有文件夹，列表头部有专门的"重扫全部"按钮。
- **主操作 / 维护工具分组**：之前 8 个按钮挤在同一行难以扫读（处理待索引、重做 OCR、清理无效记录、一键对齐文件名、重试失败 OCR、同步对比官网、添加文件夹、扫描已选文件夹）。现在主操作区只保留 3 个最常用的按钮（添加并扫描、处理待索引/OCR、↻ 同步对比官网），其余维护类按钮折叠到「维护工具 ▾」菜单里，点开才显示，菜单内每项是带描述的卡片式按钮（如"一键对齐文件名 — 把磁盘上的文件名规范化为规范命名"）。
- **批量扫描进度横幅**：扫描多个文件夹时新增蓝色横幅明确显示"正在扫描文件夹 D:\\xxx  1 / 3"，路径用 monospace 字体清晰可读，右侧 chip 显示"第几个 / 共几个"进度，再也不会出现"我也不知道现在在扫哪个"的迷茫。

## 内部改进

- **清理 8 个旧按钮 CSS 死代码**：`.index-badge` / `.process-btn` / `.requeue-mineru-btn` / `.clean-invalid-btn` / `.realign-filenames-btn` / `.sync-compare-btn` / `.add-folder-btn` / `.scan-selected-btn` 在新设计下不再被引用，连带 hover/disabled 共 ~125 行死 CSS 一次清干净。
- **修复模板 div 嵌套**：之前扫描进度/扫描结果/同步结果都误嵌在 `.search-mode-section` 内导致结构语义错乱，现在提前关闭 search-mode-section 并把这些区块归一到 `.search-section` 的直接子元素层级。

# v0.1.6

## 软件内更新流程 UX 大改进

围绕"友好方便、有反馈"，对自动更新流程做了一轮系统性优化。

- **发现新版本 / 下载完成 / 更新失败时弹 Toast 主动提示**：之前用户必须主动打开「设置 → 更新」才能看到更新状态；现在 `useUpdate` 用模块级 watch 监听 `status` 变化，关键状态切换时直接在任意页面弹 toast（"发现新版本 0.1.6，进入设置→更新下载安装" / "0.1.6 已下载完成，点立即重启完成升级" / "更新失败：xxx"），用户在规章查询、文件搜索等任何页面都能第一时间感知。
- **托盘右键菜单新增「检查更新」一键入口**：之前要 4 层（托盘→设置→更新分类→立即检查），现在右键托盘图标直接点「检查更新」就走完整 check 流程；如果发现新版本会顺便把主窗口拉出来 + 弹 toast 引导。
- **错误卡片新增「重试」按钮**：之前检查/下载失败只显示错误消息用户得手动重新点；现在记录 `lastFailedAction`，点「重试」自动判断该重新 check 还是重新 download。
- **「发现新版本」卡片显示版本对比**：标题从"有新版本可用 — 0.1.6"改为"当前 0.1.5 → 新版 0.1.6"，一眼看清升级幅度。
- **下载中显示实时速度 + 剩余时间**：进度条下面新增一行"速度: 1.2 MB/s · 剩余: 32 秒"，速度用 EMA 平滑避免数字抖动，ETA 按秒/分钟/小时自适应单位。
- **新增「跳过此版本」按钮**：用户不想升级到某个版本可以点「跳过此版本」，持久化到 `settings.update.skipVersion`；下次 check 即使后端返回该版本，前端也直接当成 UpToDate 不再打扰，直到有更高版本发布。

## 改进

- **搜索结果摘要更长**：本地搜索结果卡片里的内容摘要从 150 字符放宽到 240 字符（+60%），CSS `.card-snippet` 的 `max-height` 由 80px 同步放宽到 132px（约 7 行 12px 字号），能完整展示一段关键词附近的上下文，不再被截断到只剩一句话。

# v0.1.5

## 修复

- **下载更新过程中关闭再打开设置面板不再丢状态**：之前 `useUpdate` 的所有状态（`status`/`downloadedBytes`/`totalBytes`/`error`）都放在 composable 函数内部局部 ref，而 `SettingsPanel` 在 `App.vue` 用 `v-if="showSettings"` 包裹，关闭面板会真正卸载 `UpdateSection` → `useUpdate()` 重新初始化 → status 回到 `Idle` → 「下载并安装」按钮和进度条同时消失。Rust 端 `download_and_install_update` 继续在后台跑，但前端 listeners 已被 `onUnmounted` 解除，进度事件永远丢失。本版本把所有状态提升到模块级单例，listeners 应用生命周期内只注册一次永不卸载，设置面板关闭再重开能立即恢复真实的下载进度。

## 内部改进

- **release.ps1 加 CF 边缘缓存预热**：发布上传 + CF Purge 后，从 156 香港服务器 GET 三次 setup.exe（156 → CF HK 是同区链路，几秒内就能预热），让 Cloudflare 在亚洲 PoP 提前缓存最新版。下次大陆用户走主通道下载时直接命中边缘缓存（`CF-Cache-Status: HIT`），跳过 156 跨境出口的瓶颈链路，速度可显著提升（之前 70 KB/s 受限于晚高峰跨境出口）。

# v0.1.4

## 修复

- **修复规章卡片右侧「打开」按钮点了无反应**：之前用的是 `@tauri-apps/plugin-shell` 的 `open()`，它的默认 scope 只允许 `http(s)://` / `mailto:` / `tel:` URL，传入 Windows 文件路径会被 scope 拒绝失败；fallback 到 `revealItemInDir` 只会在文件管理器高亮文件、不会打开 PDF，且所有错误都被静默吞掉。现改为与 Ctrl+P 文件搜索一致的 `@tauri-apps/plugin-opener` 的 `openPath()`（对应 `opener:allow-open-path`，scope 已经是 `path: **` 允许任意路径），Windows 上会用系统默认 PDF 阅读器直接打开；失败时回退到在文件管理器中高亮，再失败弹 toast 说明原因，不再静默。

# v0.1.3

## 修复

- **修复右上角齿轮打开设置面板是空白的问题**：之前齿轮按钮通过 `provide/inject` 触发设置面板，存在渲染时序问题导致弹窗内部不渲染。现统一改为与系统托盘菜单完全相同的路径（emit `open-settings` 事件），保证两种入口行为一致。
- **「下载并安装」按钮不再闪现即消失**：删除发现新版本后 1 秒自动触发下载的逻辑。现在该按钮持续可见，由用户主动点击才开始下载，期间显示实时进度条（下载字节/总大小、百分比），下载完成后再由用户主动点击「立即重启」完成升级，全过程在应用内进行，不跳浏览器。

# v0.1.2

## 修复

- **修复「一键对齐文件名」失败**：v0.1.1 中点击该按钮报 `Schema error: 'Field "url" is not indexed.'`。根因是搜索索引 `url` 字段从一开始就只是 `STORED` 没有倒排索引，按 url 做的 `TermQuery` / `delete_term` 操作都会失败（实际还隐性影响 `exists` / `delete_by_urls`）。本版本把 `url` 字段升级为 `STRING | STORED`，并把索引 Schema 升到 v3。
- **无感知索引迁移**：v0.1.1 用户启动 v0.1.2 时会自动迁移搜索索引（v2→v3），从旧索引读出所有文档并用新 Schema 重写，**正文不丢、不需要重做 OCR**。3500 篇规章迁移仅需数秒。

# v0.1.1

## 新功能

- **一键对齐 PDF 文件名**：把磁盘上的 `<hash>.pdf` 历史文件批量重命名为「文号_标题.pdf」，并同步更新数据库和搜索索引。在「规章查询 → 本地索引信息」区域新增蓝色按钮触发。
- **下载/复制即用规范文件名**：从 CAAC 官网下载或从本地文件夹复制规章时，直接保存为「文号_标题.pdf」格式，不再出现长串十六进制文件名。重名自动加序号后缀。
- **清理无效 OCR 记录**：在失败 OCR 数 > 0 时显示「清理无效记录」按钮，删除磁盘已不存在或后缀非 .pdf 的残留记录（这类记录重试 OCR 永远不会成功）。

## 修复

- **修复搜索关键词高亮的崩溃**：当搜索结果包含中文、emoji 等多字节字符且关键词大小写折叠后字节长度变化时（如某些土耳其语 i、希腊语字符），高亮逻辑会越界 panic 导致整次搜索失败。改用 ASCII-only 大小写折叠，保证字节索引稳定。
- **修复内容摘要提取的同类崩溃**：`extract_snippet` 同样存在大小写折叠导致的字节错位风险，一并修复。

## 内部改进

- 抽取 `regulation::filename` 模块统一 PDF 文件名规则：非法字符替换、长度上限截断、SHA-256 兜底。
- 新增后端命令 `regulation_realign_pdf_filenames`，事务化地批量重命名磁盘文件 + 更新数据库 `file_path/url` + 同步 Tantivy 索引 `file_path` 字段。
- 新增数据库辅助函数 `delete_files_by_ids`，配合无效记录清理。
- 单元测试覆盖：UTF-8 高亮回归样本、`build_pretty_filename` 边缘场景（超长标题、全非法字符、空标题等）。
